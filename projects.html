Portfolio Website

A fast, accessible personal site that markets your skills and showcases projects with clear CTAs. Designed to pass recruiter skims in 30 seconds.

Problem & Scope

  • Many student sites are slow, inaccessible, and unclear. Objective: ship a site that loads fast on 3G and communicates value immediately.
  • Scope: Home, Projects, About, Contact, Blog index. No CMS; posts as Markdown files.

Architecture

  • Static HTML with progressive enhancement; CSS utilities; small JS for form validation and theme toggle.
  • Image pipeline (responsive srcset, lazy loading, modern formats).

Key Decisions

  • Semantic landmarks and keyboard traps audited with Axe and Lighthouse.
  • Hero content uses a 2‑line value statement + 2 CTAs (Projects, Contact).

Acceptance Criteria

  • Lighthouse: Perf ≥ 90, A11y ≥ 95, Best Practices ≥ 95.
  • CLS < 0.1, LCP < 2.5s on emulated Moto G4 / Fast 3G.
A11y ≥ 95 CLS < 0.1 LCP < 2.5s

Tasks App (MERN)

Task management with auth, filters, and server pagination. Built to demonstrate clean REST design, data modeling, and client state.

Problem & Scope

  • Design a notes/tasks tool used daily on mobile and desktop. Must handle thousands of items efficiently.
  • Scope: JWT auth (login/register), CRUD tasks, filters, search, server pagination, user profile.

Architecture

  • API: Node.js + Express; DB: MongoDB (TTL indexes for archives), Mongoose schema validation.
  • Client: React with React Query (cache, optimistic updates), URL‑based filters, error boundaries.

Key Decisions

  • Offset vs cursor pagination: cursor chosen for stability at scale.
  • Central error shape {code,message,details[]} with consistent 4xx/5xx mapping.

Blog Engine

A simple CMS: Markdown posts, categories, search, and role‑based admin. Focus on schema design, SSR, and editor workflow.

Problem & Scope

  • Writers want a fast editor and reliable publishing with search. Admin needs audit and roles.
  • Scope: writer/editor roles, drafts, publish queue, search, category filters, sitemap.

Architecture

  • Server‑rendered Node.js + Postgres; slugs + unique indexes; soft delete with deleted_at.
  • Search via trigram or full‑text; cache layer with ETag and 304 responses.

Key Decisions

  • Editor autosave every 5s with conflict resolution by updated_at checks.
  • Migrations tracked via SQL files; seed users and sample posts.

E‑commerce Starter

Transactions with idempotent webhooks, consistent totals, and admin workflows. Built to demonstrate production concerns.

Problem & Scope

  • Orders must be resilient to retries and partial failures; admins need audit trails.
  • Scope: products, cart, checkout, orders, webhook reconciliation, refunds, inventory snapshots.

Architecture

  • Backend: Node.js, SQL schema (orders, payments, events). Webhook store with unique event_id.
  • Client: React routes; guarded admin; optimistic cart with server verification.

Key Decisions

  • Idempotency keys, exponential backoff; replay‑safe webhook handler.
  • Totals 100% derived server‑side; client is display only.
